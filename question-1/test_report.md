# On-Call 助手 API 测试与性能评估报告

## 1. 测试概述
本报告汇总了对 `On-Call Assistant` 应用中三个阶段（Phase 1, Phase 2, Phase 3）的全面验证。测试涵盖了基于关键词的基础检索、具备语义理解的向量检索，以及能够自主调用工具获取 SOP 内容的智能 Agent 任务执行能力。

---

## 2. API 核心功能验证 (Functional Validation)

### Phase 1：关键词基础检索 (`/v1/search`)
目标：验证字面量匹配及其对非正文标签（如 `<script>`）的过滤能力。

| 测试请求 | 预期行为 | 实际执行结果 | 状态 |
|----------|----------|--------------|------|
| `GET /v1/search?q=OOM` | 返回 sop-001 | 成功召回 `sop-001` 等 | ✅ PASS |
| `GET /v1/search?q=故障` | 命中大部分文档 | 成功按出现频次召回 10 条 | ✅ PASS |
| `GET /v1/search?q=replication` | 忽略脚本内干扰词 | 返回空列表 `[]` | ✅ PASS |
| `GET /v1/search?q=CDN` | 返回 sop-003 与 sop-010 | 返回 `['sop-010', 'sop-003']` | ✅ PASS |

### Phase 2：语义与混合检索 (`/v2/search`)
目标：验证向量嵌入模型对口语化查询的理解深度及重排序效果。

| 测试请求 | 预期行为 | 实际执行结果 (前三项) | 状态 |
|----------|----------|------------------------|------|
| `GET /v2/search?q=服务器挂了` | sop-001 与 sop-004 靠前 | `['sop-001', 'sop-010', 'sop-004']` | ✅ PASS |
| `GET /v2/search?q=黑客攻击` | sop-005 靠前 | `['sop-005', 'sop-007', 'sop-003']` | ✅ PASS |
| `GET /v2/search?q=机器学习模型出问题` | sop-008 靠前 | `['sop-008', 'sop-009', 'sop-003']` | ✅ PASS |

### Phase 3：智能 Agent 任务执行与工具调用 (`/v3/chat`)
目标：验证 Agent 能否根据模糊提问，自主选择正确的 `readFile` 参数查阅 SOP。

| 用户提问 | 期望工具调用 | 实际工具调用记录 (Tool Arguments) | 状态 | 
|----------|--------------|--------------------------------|------|
| 数据库主从延迟超过30秒...? | readFile("sop-002.html") | `{"fname": "sop-002.html"}` | ✅ PASS |
| 服务 OOM 了怎么办？ | readFile("sop-001.html") | `{"fname": "sop-001.html"}` | ✅ PASS |
| P0 故障的响应流程是什么？ | readFile("sop-001.html") | `{"fname": "sop-001.html"}` | ✅ PASS |
| 怀疑有人入侵了系统 | readFile("sop-005.html") | `{"fname": "sop-005.html"}` | ✅ PASS |
| 推荐结果质量下降了 | readFile("sop-008.html") | `{"fname": "sop-008.html"}` | ✅ PASS |

> **评估结论**：在开启 `use_embedding_search` 辅助模式下，Agent 的决策准确率达到 100%。

## 3. 性能基准测试 (Performance Benchmarks)

针对检索引擎在高并发场景下的 QPS 与时延表现如下：

| 压测目标 | 并发用户 | 总请求量 | 平均 QPS | 平均响应时延 |
|----------|----------|----------|-----------|--------------|
| `/v1/search` | 20 | 100 | **~1014.29** | **~17.2ms** |
| `/v2/search` | 5 | 25 | **~13.62** | **~354.8ms** |

### 性能分析：
1. **Phase 1**：基于内存字典的检索展现出极高的稳定性，单次访问基本在 20ms 以内，适合高频调用。
2. **Phase 2**：由于本地调用 `SentenceTransformer` 和 `Cross-Encoder` 涉及较重的数学矩阵计算，且当前受限于 CPU 资源，性能符合预期。
3. **Phase 3**：Agent 响应时间主要取决于 LLM 的生成速度和 Tool Call 往返，由于采用 SSE 流式输出，用户端感知体验较好。

---

## 4. 鲁棒性与安全性

1. **资源防御**：已实现 `readFile` 的绝对路径校验，防止 `../` 类路径穿越攻击。
2. **并发安全性**：Agent 会话采用无状态设计，SSE 推送经过明确的 event-tag 分离，多用户同时对话不会造成工具调用结果串线。
3. **配置热更新**：`config.json` 支持实时切换后端架构参数（如启用向量导航与否），便于运维动态优化 Agent 性能。

**最终结论**：系统在三阶段均达到了 `README.md` 的所有功能与行为要求，架构设计兼顾了速度（P1）与智能（P3）。
