# Phase 2: 语义搜索与混合检索实现总结

## 1. 整体思路与目标
针对 Phase 1 纯关键词匹配所导致的“语义不通”缺陷，本阶段通过引入**向量语义搜索**结合**交叉编码器 (Cross-Encoder) 重排序**，大幅提升检索准确率。
同时引入 RRF (Reciprocal Rank Fusion) 将两路搜索结果进行融合，并借助 `pickle` 持久化向量数据库以解决启动耗时痛点。

## 2. 核心架构与技术选型
| 功能模块 | 技术或模型选型 | 设计说明 |
|------|------|------|
| **向量编码模型** | `BAAI/bge-small-zh-v1.5` | 中文语境下性价比较高的 512 维嵌入模型。 |
| **重排序模型** | `BAAI/bge-reranker-base` | 交叉编码器，专用于在召回后对 Top-N 返回值进行精排打分。 |
| **混合策略** | 分数融合算法 (RRF) | 结合 Phase 1 的关键词排位与基于语义的向量得分进行动态融合。 |
| **持久化存储** | `.pkl` 文件序列面 | 直接将向量、切片以及所需元数据落地到磁盘，以毫秒级时间还原内存状态。 |
| **模型本地化** | `models/` 目录 | 自动保存 HuggingFace 模型至本地，支持冷启动加载，不再依赖外部网络实时下载。 |

## 3. 核心业务流程
该版本引擎整合了两条链路。第一部分为初始化与缓存管理：系统通过识别本地的 `vector_store.pkl` 决定是否需要从头切片与推理；第二部分为真正的查询链路，包括双路并发召回、RRF 融合与重排。
1. **持久化加载/全量切片**：应用尝试读取 `pkl`；若遇新文档，则分割长文本，送入 BGE 计算 embedding 并增量写入持久文件。
2. **双路召回**：接收查询，同步进行 Keyword Search (获取 kw_rank) 以及 Vector Search (采用 Top-K 平均分机制生成 sem_rank)。
3. **联合重排 (RRF + Reranker)**：先依赖 RRF 算法得到 Top-5 候选集合。接着对这些候选集采用“标题增强策略”（重复标题三次拼接于前部），输入 Cross-Encoder 产生最终相似分。

```text
[启动加载] --> [读取 pkl 缓存] --(存在)--> [直接恢复向量]
                      |
                   (不存在)
                      |
                      v
[HTML解析/文本切片] --> [BGE模型推理] --> [更新增量并保存 pkl]

[查询请求: /v2/search]
      |
      +--> [关键词检索] --> [kw_rank] ----+
      |                                 |
      +--> [向量搜索]   --> [sem_rank] ---+--> [RRF融合排序] --> [Top-5 重排] --> [结果]
```

## 4. 关键实现与验证结果
通过**标题信号增强**与 **Top-K Mean Pooling** 的结合机制，圆满解决了基于 Max-Pooling 时产生的“偶然单句高相似”导致的局部偏见。
关键验证场景：
- `机器学习模型出问题` -> 语义召回 `sop-008.html`。
- **本地化加速**：模型文件已下载并保存至 `models/` 下。冷启动加载时间由分钟级（网络下载）降至秒级（本地 IO）。
- **API 规范化**：严格遵循 `README.md` 返回值要求，去处了内部 `rrf_score` 字段，确保前端及测试套件的一致性评分。

## 5. 缺点与不足总结
尽管相较第一版有了质的飞跃，仍存在部分局限可以进一步推敲：
- **重排序耗时过大**：Cross-Encoder 模型极为沉重，若候选集数量过大，或部署于无 GPU 加速的纯 CPU 容器环境时，极易造成请求严重的超时阻塞。
- **固化的融合比例**：利用静态常数权重作为 RRF 的混合依据缺乏自适应能力。在极端短词搜索下可能弱于字面匹配。
- **切分粒度粗放**：一刀切的 400 字符分块可能会在逻辑结构边界（例如在步骤列表的正中央）残损语义，导致跨块语境断裂。
