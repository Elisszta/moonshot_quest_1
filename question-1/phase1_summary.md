# Phase 1: 基于关键词的基础搜索引擎实现总结

## 1. 目标与需求
本阶段的目标是实现一个基于纯关键词的文档搜索引擎，作为后续高级功能（如语义搜索与 On-Call 助手）的基础设施平台。
主要需求包括：
- 支持录入并解析 HTML 文档，提取标题和有效纯文本内容（屏蔽 `<script>`、`<style>` 等无用标签）。
- 基于关键词对解析后的文档内容执行文本匹配搜索。
- 根据命中关键词的频率实现基础相似度打分并返回带上下文的片段 (Snippet)。
- 提供简单直观的前端搜索页面进行交互。

## 2. 核心架构与技术选型
为保证良好的可维护性并为后续 Phase 的扩展打下基础，代码进行了合理的模块化拆分：
- **Web 框架**：FastAPI，负责提供异步的高性能 API 服务。
- **HTML 解析**：BeautifulSoup4，用于快速准确地从复杂的 HTML SOP 中提取结构化文本和标题。
- **模板引擎**：Jinja2，绑定 FastAPI 实现 `/v1/` 命名空间下的页面渲染。
- **依赖管理**：基于 `uv` 构建运行环境。

代码主要模块：
- `main.py`：应用入口，加载依赖并触发生命周期内的预处理（如加载文档集），同时挂载不同版本的路由集。
- `api/v1/router.py`：定义 Phase 1 的各项端点，如 `/documents`, `/search`，以及前端页面呈现。
- `services/search_engine.py` & `services/document_store.py`：基于内存倒排或遍历检索的基础搜索逻辑及简单打分机制。

## 3. 核心业务流程
当系统接收到一个长文本查询请求时，将执行以下的流转处理：
1. **数据加载阶段**：系统启动时，由 `document_store` 扫描预设的 `data/` 目录，利用 `html_parser` 精洗出所有干净的纯文本。
2. **搜索触发阶段**：用户通过 GET 请求提交包含查询词 `q` 的参数。
3. **匹配与打分阶段**：搜索引擎遍历缓存在内存中的纯文本内容，根据关键字出现的频次进行命中验证和简单打分排序。
4. **结果组装阶段**：为命中的文档提取对应上下文内容，封装为 Snippet 连同元数据一起返回。

```text
[系统启动] --> [扫描 data/ 目录] --> [解析 HTML 提取文本] --> [构建内存文档商店]
                                                                    |
                                                                    v
[用户检索请求: /v1/search] --------------------------------------> [关键词遍历匹配]
                                                                    |
                                                                    v
[返回 JSON 结果/前端渲染] <---------- [提取上下文 Snippet] <---------- [计算命中频率得分]
```

## 4. 关键实现与验证结果
全部功能顺利完成并在本地通过了多项基础测试：
- `GET /v1/search?q=OOM`：正确返回了与内存溢出排查相关的 `sop-001` 等文档。
- `GET /v1/search?q=故障`：匹配到多个包含该词的文件，且能按照频率高低形成初步排序。
- **特殊标签与字符处理**：成功忽略了 `<script>` 等标签中的误导词汇（如 replication）；支持 URL 编码方式查询特殊字符（如 `&`）。

## 5. 缺点与不足总结
尽管完成了基础目标，本阶段的系统仍存在以下明显的缺点：
- **语义理解缺失**：仅支持字面级别的“硬匹配”，无法领会“服务器挂了”等口语化表述，同义词或近义词将导致漏召回。
- **性能与持久化瓶颈**：文档保存在纯内存中，当 SOP 库逐渐增大时，每次重启都需重新解析数十个 HTML 文件，导致服务冷启动异常缓慢。
- **排序策略单一**：仅仅依赖 TF（词频）的简单打分算法容易被重复堆砌词汇的文档影响而失去公允，更缺乏 IDF 或文档整体权重的综合考量。
