# Phase 3: On-Call 助手 Agent 实现总结

## 1. 目标与需求
本阶段旨在基于已有的数据支持，落实一个具备完整交互能力的 On-Call 问答大模型。
Agent 被设定了严格的能力边界，它无法主动搜索全网页或列出目录结构，**仅被授予了一个可以指定精确路径的文件阅读工具 (`readFile`)**。通过该工具，Agent 应当利用大模型自身的逻辑推理能力，先精准调研所需排版信息，而后融合并输出完整解决方案。

## 2. 核心架构与技术选型
采用纯异步处理机制并利用大模型的 Tool Calling 特性，通过长轮询（或流式长链接）向前端传递丰富状态信息。
- **服务端通信**：基于 FastAPI 的 `StreamingResponse` 提供 Server-Sent Events (SSE) 流协议，向客户端精准推送每个事件。
- **大模型生态**：引入官方最新的原版 `openai` SDK (`AsyncOpenAI`) 以兼容各种使用 OpenAI 的代理分发服务。
- **前端可视化**：HTML + 原生 JS + TailwindCSS，前端针对推流数据类型自动解析，提供对话加载以及中间形态 Tool 调用（如“正在读取…”，“已读取”）的气泡更新。
- **动态配置**：通过根目录的 `config.json` 即时切换底层使用的大语言模型、API 密钥与连接端点。

## 3. 核心业务流程
一轮经典的带有工具调用的会话流程包括：用户提交、服务端环境判断及模型调用、工具拦截执、内容补充回填至最终推流输出。
1. **请求封装与意图识别模式**：针对大模型“盲猜”文件名极其困难的缺陷，提供向量辅助模式：在向 LLM 会话发起之前，系统在后台使用 Phase 2 的检索模块提取相关的 Top-N 文档标题列表，作为隐性提示注入 System Prompt，诱导其产生针对该些文件的阅读动机。
2. **第一阶段推理 (Function Calling)**：大模型分析当前问题与提示词，在返回体中抛出希望执行的工具调用指令（如 `readFile(name: "sop-002.html")`），同时触发 SSE 下发状态 `tool_call` 给前端。
3. **本地工具接管执行**：后端拦截截断此调用，执行本地真实函数 `read_file()`，读取磁盘内容（同时预防路径穿越攻击），然后伪装成一并 `tool_result` 返回对话序列。
4. **流式整合报告**：再次向 LLM 发起提交刚才加载到的具体步骤长文本，模型随之开始流式地 (chunks) 向前端打印它结合排故障现场得出的指导。

```text
[用户发送指令] --> [向量辅助导航?] --(是)--> [V2 Search 提取名单注入提示词]
                              |
                            (否)
                              |
                              v
[发起初次 LLM 请求] --> [判断需查阅 SOP?] --(是)--> [SSE 下发 tool_call 至前端]
                              |                      |
                            (不需要)                 v
                              |              [执行 read_file 并回填对话流]
                              v                      |
[下发消息流式 chunks] <--- [再次请求 LLM 总结内容] <---+
```

## 4. 关键实现与验证结果
系统运行极其顺畅，主要验证里程碑包括：
- **实时事件推流效果**：舍弃笨重的 LangChain 抽象，凭借原生的 `AsyncOpenAI` 和精细的状态管理，做到了将底层复杂运行步骤（从 Thought 到 Tool Execute 到 Final Respond）完全可视化反映至前端。
- **文件发现率提升**：单纯依靠 `readFile` 工具的前提下，结合向量支持的 System 拦截机制促使 Agent 查阅文件的成功与匹配率大幅度趋向 100%。

## 5. 缺点与不足总结
该高级 Agent 雏形当前阶段仍然面临一些隐患与瓶颈：
- **Token 上下文溢出风险**：由于只是单纯的拿大块 SOP `read_file` 直接硬塞进上下文里。即使排版精美，一旦碰到巨型 HTML 文件，也面临 Token 超限或大模型中途遗忘信息的风险，尚未实现更精细的内部重分块或截断策略。
- **受限于大模型的 Tool 支持能力**：目前高度依赖类似于 GPT 等对 Function Calling 经过精调的模型表现。若是切换至较差的开源平替，常常会出现强行输出伪造工具调用或者格式错乱的情况导致全盘崩溃。
- **没有错误自动恢复机制**：目前流程偏向线性单向链，假若 Agent 真的猜错读错了与其完全无关的文档，或是读取到无效结构，当前尚未设立“校验并重新自主抉择寻找新文档”的闭环反思重试逻辑。
